# テックブログ清書スキル

blog-brainstorm で作成した構成メモ（HTMLコメント）付きの記事ファイルを元に、ユーザーと対話しながらセクションごとに本文を書き上げるスキル。

## 基本方針

このスキルの役割は「共同執筆パートナー」。ユーザーの経験や知見を引き出しながら、一緒に記事を書いていく。

テックブログの価値は「身体性」— 実際に手を動かした経験、つまずいたポイント、理解できた瞬間の記録にある。Claude が一般的な解説を書くのではなく、ユーザーの体験を文章として形にすることに集中する。

## 清書の進め方

### Step 1: 記事ファイルの確認

まず対象の記事ファイルを読み、構成メモ（HTMLコメント）を把握する。ユーザーにどのセクションから書き始めるか確認する。

特に指定がなければ、上から順に進める。ただし「はじめに」は全体が見えてから書いた方が良いことが多いので、本論のセクションから始めることを提案してもよい。

### Step 2: セクションごとの対話と執筆

各セクションで以下の流れを繰り返す：

**1. 経験を引き出す質問をする**

構成メモの内容を見て、本文を書くために足りない情報を質問する。特にユーザーの実体験に関わる部分を重点的に聞く。

質問の例：
- 「このセクションで、実際にどんな操作をした？」
- 「エラーメッセージや画面のスクショはある？」
- 「この選択をした決め手は何だった？」
- 「他の選択肢も検討した？なぜこれにした？」
- 「ここで一番伝えたいことを一言でいうと？」

質問は1〜2個ずつ。矢継ぎ早に聞かない。

**2. 回答を元に本文を書く**

ユーザーの回答が得られたら、そのセクションの本文を書いて記事ファイルに反映する。

執筆のルール：
- ユーザーの言葉をできるだけ活かす。過度に整えて「AI が書いた感」を出さない
- 文体は「です・ます」調で統一する。経験に基づく部分も「〜しました」「〜でした」で書く
- 一般的な説明が必要な箇所は、ユーザーの経験と絡めて書く（「公式ドキュメントによると〜だが、実際に試すと〜だった」など）
- 断言を避け「私のケースでは」「〇〇の条件下では」と限定する
- コードブロック、コマンド例、設定例は具体的に書く
- 参考リンクは構成メモに含まれていればそのまま使う

## 文体・トーンのガイドライン

以下の4つの要素を意識して書く。

### tl;dr 先出し
記事冒頭（「はじめに」の直後）に3〜5箇条の要点を置く。結論を先に出して読者を引き込む。忙しい読者が要点だけ拾えるようにする。

### 口語混入
「めっちゃ」「シュッと」「面倒くさい」「消耗した」「〜っぽい」「なんか」など、話し言葉を適度に混ぜる。読者との距離を縮める効果がある。ただし全文が口語になるとチャラくなるので、要所で挟む程度にする。

### 率直な限界開示
「もうちょい研究が必要」「妥協の産物になった」「使う場合は自己責任で」など、できないこと・課題・不確実な部分を隠さず書く。信頼性が上がる。

### 具体性とコード
抽象論の合間に必ず具体的なコマンド / コード / 数値 / スクショを挟む。「動きました」の事実報告を大事にする。

**3. ユーザーに確認する**

書いた内容をユーザーに見てもらい、フィードバックを反映する。「しっくりこない」と言われたら、何が違うのかを聞いて調整する。

### Step 3: 仕上げ

全セクションが書き終わったら：

1. **「はじめに」を書く**: 全体が見えた状態で導入文を書く。問題提示 → 解決予告 → 信頼の根拠の流れを意識する
2. **「まとめ」を書く**: 記事全体の振り返り
3. **タイトルの最終確認**: 本文の内容とタイトルが合っているか確認する
4. **通し読みの提案**: 全体を通して読んでもらい、最終調整する

### Step 4: 公開前チェック

以下を確認する：
- frontmatter の topics が適切か
- コードやコマンドの動作確認が取れているか（ユーザーに確認）
- 機密情報が含まれていないか
- 参考リンクが正しいか

textlint は hooks で自動実行されるため、文章校正はそちらに任せる。

## 対話のスタイル

- セクション単位で進める。一気に全部書こうとしない
- 「書けるところから書く」でOK。順番にこだわらない
- ユーザーが「ここは自分で書く」と言ったらそのセクションはスキップする
- 80%の完成度で公開する前提。完璧を求めない
- 図や画像の挿入が必要な箇所は、プレースホルダーとして `![説明](画像パス)` を置いてユーザーに後で差し替えてもらう
